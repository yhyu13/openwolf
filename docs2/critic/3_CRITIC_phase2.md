### Critical Issues

#### 1. Parent matching bug in the spec (Phase 2.3)

```typescript
case "parent":
  return eventPaths.some(ep =>
    cue.path.some(cp => cp.startsWith(ep))  // BUG: this is BACKWARDS
  ) ? 0.7 : 0.0;
```

The code says `cp.startsWith(ep)` — checking if the cue path starts with the event path. But "parent match" should mean: given a file path, find events for its parent directories. So if cue is `src/auth/middleware.ts`, a parent match should find events where event paths start with the cue path's parent segments (e.g., `src/auth/`, `src/`).

The logic as written checks the **opposite direction**: it looks for cue paths that start with event paths. This is D4 ("Location parent matching incorrect") — but the bug is in the spec itself, not just the implementation. Anyone implementing from this spec will write the wrong logic.

**Fix should be**: `ep.startsWith(cp)` — for each event path, check if it starts with a parent of the cue path.

#### 2. Semantic similarity weight (30%) has no implementation

The spec allocates 30% of scoring to "semantic similarity" defined as "Entity/tag overlap with query". But no code is shown for:
- Extracting entities from a question cue
- Computing overlap between query entities and event tags/entities
- The Phase 2.3 section explicitly defers this to Phase 3 with "simple tag matching"

This means **30% of the scoring algorithm is undefined**. If this stays in Phase 2, implementation details must exist. If it moves to Phase 3, the weight must be redistributed or the feature disabled.

---

#### 3. Index update strategy risks index drift

Phase 2.1 says:
- "Real-time: On every `addEvent()`, update index in-memory"
- "Batch persist: Every 10 events, write index to disk"

This means disk persistence lags in-memory by up to 9 events. If the process crashes between batch writes, the index rebuild will miss those events (though they survive in the event store). This is the D3 defect ("Index not persisting batch-wise"). The spec documents the behavior but doesn't acknowledge it as a durability trade-off.

---

#### 4. `penalty` and `reward` valence source undefined

Phase 1 valence detection only produces `neutral` or `trauma`. The `penalty` and `reward` valences are defined in types but never generated by the hook wiring. Phase 2.5 enhances `pre-write.ts` to warn about penalized files, but:

- Where do `penalty` events come from?
- The `pre-write` hook currently does nothing in Phase 1
- `post-write` hook generates events but only `neutral` or `trauma`

Without a valence generation strategy, the Phase 2 pre-write penalty warning will never fire.

---

#### 5. `StateCue.recent_valence` has no population site

```typescript
interface StateCue {
  recent_valence: Valence[];
}
```

This requires the caller to track and pass in the recent valence history across turns. But:
- The hooks don't maintain session state between calls
- The CLI doesn't track valence history
- No session-level context object exists in the current architecture

The `recent_valence` field would always be empty unless a session-level context manager is introduced — which is not in the Phase 2 spec.

Phase 2.5 modifies `pre-write.ts` to emit warnings before edits. This changes hook behavior that users may depend on. T12.1 tests the new warning but does not test:

D3 ("Index not persisting batch-wise") is listed as a defect target to fix in Phase 2. T12.3 ("cue-index persists across sessions") tests the persistence behavior. This is circular — the test verifies the fix, but doesn't explain how D3 was discovered or whether it was a pre-existing issue or a Phase 2 prediction.

#### 3. T10.11 "Error type matching works" has no concrete steps

The test is named but has no validation steps. No test of:

Same gap for T10.12 (recent_valence detection).

#### 4. T11 recall CLI tests lack expected output
